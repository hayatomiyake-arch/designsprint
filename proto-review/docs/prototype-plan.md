# プロトタイプ計画 — 3パターン比較（v2）

## 目的
**サービス体験の方向性**を3パターンのコンセプトで発散し、
どの体験が最も価値があるかを検証してから収束させる。

> v1からの変更点: ストーリーボード型 → **実際のツール体験モック型**に変更。
> 「自分のプロトタイプを横に見ながら、何が足りないかわかる」体験を全パターン共通の前提とする。

---

## ステップ

```
Step 1: コンセプト（ツール体験モック）— 3パターン発散
  ↓ どの見せ方が最も「使いたい」と感じるか？
Step 2: 収束 → 選定コンセプトのコア体験プロトタイプ
  ↓ 操作感・情報設計を検証
Step 3: MVP
```

---

## 共通の前提

- **デモ画面が常に表示される**: ユーザーのプロトタイプ（サンプルとしてログイン画面）が常に見えている状態でレビューを体験
- **デザイナーでなくても使える**: 専門用語ではなく、視覚的に「何が問題か」がわかる
- **Figma Makeの「生成+デモ」のように、「評価+デモ」を並べて見られる**

---

## Pattern A: 「赤ペンレビュー」

### 一言で
> プロトタイプ画面の上に直接「赤ペン」でマーカーが入る — 問題箇所が一目瞭然

### コア体験
画面を見ている → 問題のある箇所に番号付きマーカーが表示される → マーカーをクリックすると具体的な指摘と改善案が右パネルに出る

### 非デザイナーにとっての価値
- 「画面のここが問題」が**位置で**わかる
- 赤ペンの感覚は誰でも馴染みがある（学校の添削と同じ）
- 全体のスコアではなく**具体的な場所**にフォーカス

### UI構成
```
┌─────────────────────────────────────┐
│  [Proto Review — 赤ペンレビュー]     │
├──────────────┬──────────────────────┤
│              │  ❶ コントラスト不足   │
│   📱 デモ画面 │  ━━━━━━━━━━━━━━━   │
│   (iframe)   │  説明文...           │
│              │  💡 改善案: ...       │
│  ❶ ❷ ❸     │                      │
│  (マーカー)   │  ❷ ボタンが小さい    │
│              │  ━━━━━━━━━━━━━━━   │
│              │  説明文...           │
├──────────────┴──────────────────────┤
│  レビュー実行  |  別のプロトを見る     │
└─────────────────────────────────────┘
```

### 賭けているポイント
- **場所の特定**が最大の価値 — 「どこ」がわかれば「何を直すか」は自然にわかる
- 赤ペン添削のメタファーは**非デザイナーにも直感的**

### リスク
- AIが正確な位置を指せるか（HTMLの構造解析精度）
- マーカーが多すぎると逆にわかりにくい

---

## Pattern B: 「Before / After」

### 一言で
> 今の画面と改善版を並べて見比べる — 「こう変えるとこう良くなる」が目でわかる

### コア体験
左に自分の画面 → 右にAIが提案した改善版 → スライダーで重ねて比較 → 下に「何が変わったか」の解説

### 非デザイナーにとっての価値
- 言葉で説明されなくても**2つ並べれば違いが見える**
- 「良い」と「改善前」の差を体感できる
- 改善の方向性が具体的なビジュアルで示される

### UI構成
```
┌───────────────────────────────────────┐
│  [Proto Review — Before / After]       │
├──────────────────┬────────────────────┤
│                  │                    │
│  📱 あなたの画面   │  📱 改善イメージ    │
│  (現在のHTML)     │  (AI提案版)        │
│                  │                    │
│     BEFORE       │     AFTER          │
│                  │                    │
├──────────────────┴────────────────────┤
│  ◀━━━━━━━━●━━━━━━━━▶ スライダー比較   │
├───────────────────────────────────────┤
│  変更点:                               │
│  ✦ ボタンの背景色 #ccc → #1a73e8      │
│  ✦ 余白 8px → 24px                    │
│  ✦ フォントサイズ 14px → 16px          │
└───────────────────────────────────────┘
```

### 賭けているポイント
- **ビジュアルの比較**は言葉より強い — 100の説明より1の見比べ
- 改善が「抽象的なアドバイス」ではなく「具体的な画面」で示される

### リスク
- AI生成の改善版の品質（悪い改善案だと逆効果）
- 「AIが勝手に変えた」感が出て、自分の学びにならない？

## Pattern C: 「デザイン健康診断」

### 一言で
> チェック項目を一つずつON/OFFしながら、画面上の問題箇所がハイライトされる"健康診断"

### コア体験
チェック項目リスト（コントラスト、余白、タップ領域…）→ 項目をONにすると画面上の該当箇所がハイライト → OK/NGが明確に色分け

### 非デザイナーにとっての価値
- **「何を見ればいいか」のチェックリスト**が明確
- 赤=NG、緑=OKの色分けで判断に迷わない
- 一つずつ確認できるので情報過多にならない

### UI構成
```
┌───────────────────────────────────────┐
│  [Proto Review — デザイン健康診断]       │
├──────────────┬────────────────────────┤
│              │  診断項目              │
│   📱 デモ画面 │                        │
│   (iframe)   │  ☑ コントラスト  🔴 NG  │
│              │    → 4.2:1 (基準4.5)   │
│  [ハイライト  │                        │
│   オーバーレイ]│  ☑ タップ領域   🟢 OK  │
│              │    → 48px (基準44px)   │
│              │                        │
│              │  ☐ 余白の統一   —      │
│              │  ☐ 視覚的階層   —      │
├──────────────┴────────────────────────┤
│  総合: 2/4 クリア  ━━━━━━░░░░ 50%     │
└───────────────────────────────────────┘
```

### 賭けているポイント
- **健康診断のメタファー**は誰にでも馴染みがある
- 一項目ずつ見るので、**認知負荷が低い**
- 「全項目クリア」が明確なゴールになる

### リスク
- チェック項目が固定的すぎて、創造性のある問題を見逃す？
- 項目数が少ないと物足りない、多いと面倒

## 3パターン比較（v2）

| 観点 | A: 赤ペンレビュー | B: Before / After | C: デザイン健康診断 |
|---|---|---|---|
| **核心の問い** | 「どこが問題？」 | 「どう良くなる？」 | 「基準を満たしてる？」 |
| **非デザイナー向け** | 場所でわかる | 見比べでわかる | チェックリストでわかる |
| **情報の密度** | 中（マーカー数次第） | 高（画面2つ＋差分） | 低（一項目ずつ） |
| **デモ画面の使い方** | 上にマーカーを重ねる | 横に並べて比較 | ハイライトを切替 |
| **ユーザーの感情** | 「ここか！」 | 「こんなに変わるのか」 | 「あと2つ直せばOK」 |
| **技術的な難しさ** | 中（要素位置の特定） | 高（改善版HTML生成） | 低（ルールベース判定可） |
| **AI依存度** | 中 | 高 | 低（ルールベースで可能） |

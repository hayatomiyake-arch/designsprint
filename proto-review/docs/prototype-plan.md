# プロトタイプ計画 — 3パターン比較

## 目的
コア体験の方向性を3パターンのプロトタイプで検証し、A/Bテスト的に最適なアプローチを選定する。

---

## ステップ

```
Step 1: コンセプト（ストーリーボード）プロトタイプ — 3パターン
  ↓ どのコンセプトが最も刺さるか検証
Step 2: コア体験プロトタイプ — 選定パターンを深堀り
  ↓ 操作感・情報設計を検証
Step 3: MVP
```

---

## Pattern A: 「プレビューギャラリー型」

### コンセプト
プロトタイプをカード形式で一覧表示。選択するとプレビュー + レビューパネルが開く。
**Figma のファイル一覧 + インスペクトパネルのイメージ。**

### ストーリーボード
```
[1] ギャラリー画面
┌────────────────────────────────┐
│  Proto Review                  │
│ ┌──────┐ ┌──────┐ ┌──────┐   │
│ │ Card │ │ Card │ │ Card │   │
│ │ 📱   │ │ 📱   │ │ 📱   │   │
│ │ name │ │ name │ │ name │   │
│ └──────┘ └──────┘ └──────┘   │
│         [+ 追加]              │
└────────────────────────────────┘

[2] プレビュー + レビュー画面
┌────────────────────────────────┐
│ ← 戻る    prototype-name      │
│ ┌──────────┐ ┌──────────────┐ │
│ │          │ │ Review       │ │
│ │  Phone   │ │ Score: 72    │ │
│ │  Frame   │ │ Good: ...    │ │
│ │  Preview │ │ Dev: ...     │ │
│ │          │ │              │ │
│ └──────────┘ │ [改善ヒント]  │ │
│              └──────────────┘ │
└────────────────────────────────┘
```

### 特徴
- 直感的。見れば使い方がわかる
- プロトタイプの全体像が一目でわかる
- レビューは個別表示

### 検証ポイント
- カード一覧で十分な情報量があるか？
- プレビューとレビューの2画面構成は面倒でないか？

---

## Pattern B: 「ワークスペース型」

### コンセプト
1画面にプレビュー・コンテキスト入力・レビュー結果を統合。
**VS Code のエディタ + サイドパネルのイメージ。**

### ストーリーボード
```
[1] ワークスペース画面（メイン）
┌────────────────────────────────────────┐
│ Proto Review                           │
│ ┌────┐ ┌──────────┐ ┌──────────────┐  │
│ │List│ │          │ │ Context      │  │
│ │ p1 │ │  Phone   │ │ 誰向け: ___ │  │
│ │ p2 │ │  Frame   │ │ 目的: ___   │  │
│ │ p3 │ │  Preview │ │              │  │
│ │    │ │          │ │ [レビュー実行]│  │
│ │    │ │          │ ├──────────────┤  │
│ │    │ │          │ │ Review結果   │  │
│ │[+] │ └──────────┘ │ Good / Dev   │  │
│ └────┘              └──────────────┘  │
└────────────────────────────────────────┘
```

### 特徴
- 画面遷移なし。すべてが1画面に収まる
- コンテキスト入力とレビューが隣接し、因果が明確
- プロトタイプの切り替えはサイドバーで瞬時

### 検証ポイント
- 情報が詰まりすぎて認知負荷が高くないか？
- モバイル対応が難しくないか？

---

## Pattern C: 「ステップ型（ウィザード）」

### コンセプト
「選ぶ → コンテキスト入力 → レビュー → 改善」を段階的に案内。
**フォームウィザード / オンボーディングフローのイメージ。**

### ストーリーボード
```
[Step 1] プロトタイプ選択
┌────────────────────────────────┐
│ Step 1/4: プロトタイプを選択   │
│                                │
│ ┌──────┐ ┌──────┐ ┌──────┐   │
│ │ 📱   │ │ 📱   │ │ 📱   │   │
│ └──────┘ └──────┘ └──────┘   │
│                                │
│              [次へ →]          │
└────────────────────────────────┘

[Step 2] コンテキスト入力
┌────────────────────────────────┐
│ Step 2/4: レビューの前提を入力 │
│                                │
│ このUIは誰向け？  [________]  │
│ 何のため？        [________]  │
│ 特に気になる点は？ [________]  │
│                                │
│    [← 戻る]    [レビュー開始]  │
└────────────────────────────────┘

[Step 3] レビュー結果
┌────────────────────────────────┐
│ Step 3/4: レビュー結果         │
│ ┌──────────┐                  │
│ │  Preview  │  Score: 72      │
│ │          │  Good: ...      │
│ └──────────┘  Dev: ...       │
│                                │
│    [← 戻る]    [改善する →]   │
└────────────────────────────────┘

[Step 4] 改善 & 再レビュー
┌────────────────────────────────┐
│ Step 4/4: 改善して再レビュー   │
│ ┌─────┐ → ┌─────┐            │
│ │Before│   │After │            │
│ │ 72  │   │ ??  │            │
│ └─────┘   └─────┘            │
│                                │
│ [HTMLを貼り替え] [再レビュー]  │
└────────────────────────────────┘
```

### 特徴
- 迷わない。次に何をすべきか常に明確
- 初心者に優しい。ステップごとに集中できる
- 改善ループが構造化されている

### 検証ポイント
- ステップ数が多くて面倒に感じないか？
- 慣れたユーザーにはまどろっこしくないか？

---

## 比較まとめ

| 観点 | A: ギャラリー型 | B: ワークスペース型 | C: ステップ型 |
|---|---|---|---|
| 直感性 | ◎ 見ればわかる | ○ 慣れれば高速 | ◎ 迷わない |
| 情報密度 | ○ 適度 | △ 高密度 | ◎ ステップ分散 |
| 操作効率 | ○ 2画面遷移 | ◎ 遷移なし | △ ステップ遷移 |
| モバイル対応 | ◎ カード向き | △ 3カラム困難 | ◎ 縦フロー |
| 初心者向け | ○ | △ | ◎ |
| パワーユーザー | ○ | ◎ | △ |

# プロトタイプ計画 — 3パターン比較

## 目的
**サービス体験の方向性**を3パターンのコンセプトで発散し、
どの体験が最も価値があるかを検証してから収束させる。

> UIレイアウトの違いではなく、「何を体験するサービスなのか」のレベルで振り切る。

---

## ステップ

```
Step 1: コンセプト（ストーリーボード）— 3パターン発散
  ↓ どの体験が最も刺さるか？
Step 2: 収束 → 選定コンセプトのコア体験プロトタイプ
  ↓ 操作感・情報設計を検証
Step 3: MVP
```

---

## Pattern A: 「辛口デザインクリティック」

### 一言で
> プロトタイプを見せたら、忖度なしの本気フィードバックが返ってくる"師匠"

### コア体験
プロトタイプを投げ込む → **「ここがダメ、理由はこう、こう直せ」** という厳しく具体的なフィードバックを受ける

### ストーリーボード（ユーザーの行動と感情）
```
[1] 投げ込む
   ユーザー: Claude Codeで作ったログイン画面のHTMLをペースト
   感情: 「まあまあ良くできたかな」

[2] 師匠の一撃
   AI: 「コントラスト比が4.2:1でAAを満たしていない。
        主要CTAが視覚的階層の3番目に埋もれている。
        エラー状態が未定義。」
   感情: 「うっ…でも確かにそうだ」

[3] 具体的な処方箋
   AI: 「CTAのフォントウェイトを600→700に、
        周囲に24px以上の余白を確保。
        これだけで視覚的優先度が1段上がる。」
   感情: 「なるほど、やってみよう」

[4] 改善して再提出
   ユーザー: 指摘を反映したHTMLを再ペースト
   AI: 「CTAの存在感は改善。ただしフォーム全体の
        垂直リズムが崩れた。行間を再調整せよ。」
   感情: 「もっと良くなる余地があるのか…！」
```

### このパターンの賭けているポイント
- **「正直さ」に価値がある** — 周囲に「良いんじゃない？」としか言わない人しかいない
- **即座に具体的な改善アクションが得られる** — 抽象的な感想ではなく処方箋
- **繰り返し提出する中毒性** — 「もう1回見てもらおう」のループ

### リスク
- 厳しすぎてモチベーションが下がる？
- フィードバックの信頼性（AIが間違った指摘をしたら一気に冷める）

## Pattern B: 「プロトタイプ進化録」

### 一言で
> 自分のプロトタイプがどう進化してきたかを可視化する"成長日記"

### コア体験
プロトタイプを登録するたびに自動記録される → **時系列で「Before → After」の変化を眺める** → 自分の成長を実感する

### ストーリーボード（ユーザーの行動と感情）
```
[1] 最初のプロトタイプを登録
   ユーザー: TODOアプリのv1をペースト
   システム: スナップショットを保存。初期スコア: 58
   感情: 「まずは記録、と」

[2] 数日後、改善版を登録
   ユーザー: TODOアプリのv2をペースト
   システム: v1→v2の差分をハイライト。スコア: 58→71
            「色のコントラスト改善 (+8)、余白の統一 (+5)」
   感情: 「おお、ちゃんと良くなってる」

[3] タイムラインを眺める
   システム: v1 → v2 → v3 が並んで表示される
            進化のグラフ、改善ポイントの累積
   感情: 「1ヶ月前の自分、こんなの作ってたのか…」

[4] 別のプロトタイプにも応用
   ユーザー: 新しいプロジェクトのUIを登録
   システム: 過去の学びを横断表示
            「以前のTODOアプリでも同じ余白の問題がありました」
   感情: 「自分のクセが見えてきた」
```

### このパターンの賭けているポイント
- **「成長の可視化」に価値がある** — デザインスキルの向上は普段見えない
- **記録が資産になる** — 使えば使うほどデータが貯まり、離れられなくなる
- **自分のクセ・パターンの発見** — 横断的なフィードバックで気づきが生まれる

### リスク
- 複数バージョンを登録するモチベーションが続くか？
- 最初の1回だけで終わってしまわないか？
- 「記録すること自体」に価値を感じるか、面倒に感じるか？

## Pattern C: 「出荷OKライン」

### 一言で
> 「これ、人に見せて大丈夫？」の不安を解消する"品質チェックゲート"

### コア体験
プロトタイプを通すと **「出荷OK / もう少し / やり直し」の判定** が出る → 基準をクリアするまで改善 → 自信を持って共有できる

### ストーリーボード（ユーザーの行動と感情）
```
[1] チェックに通す
   ユーザー: クライアントに見せたいLPのプロトタイプを投入
   感情: 「これで良いのか正直わからない…」

[2] 判定が出る
   システム: ━━━━━━━━━━░░ 72/100
            判定: 「もう少し」
            未クリア項目:
            ☒ モバイル時のタップターゲットが小さい (44px未満)
            ☒ 主要動線が2クリック以上
            ☑ 色のコントラスト — OK
            ☑ 視覚的階層 — OK
   感情: 「あと2つ直せばいいのか、明確！」

[3] 指摘項目を潰す
   ユーザー: タップターゲットを48pxに修正、導線を短縮
   システム: ━━━━━━━━━━━━ 88/100
            判定: 「出荷OK」 ✓
   感情: 「よし、自信を持って見せられる！」

[4] バッジ/証明として活用
   システム: 「Proto Review: Quality Passed」のバッジを生成
   ユーザー: プロトタイプと一緒に共有
   感情: 「客観的なお墨付きがあると安心」
```

### このパターンの賭けているポイント
- **「不安の解消」に価値がある** — 「これで大丈夫？」は誰もが持つ悩み
- **明確なゴールがある** — 「OK」が出るまで頑張れるゲーム性
- **チェックリストが具体的** — 何を直せばいいか迷わない

### リスク
- 「OK」の基準がAIの判定で十分信頼できるか？
- チェック項目が画一的で、創造性を殺さないか？
- 「バッジ」に実際の価値はあるか？

---

## 3パターン比較

| 観点 | A: 辛口クリティック | B: 進化録 | C: 出荷OKライン |
|---|---|---|---|
| **コアの価値** | 正直で具体的なフィードバック | 成長の可視化 | 不安の解消・品質保証 |
| **ユーザーの感情** | 「悔しいけど的確」 | 「成長してる実感」 | 「自信を持てた」 |
| **利用頻度** | 高（毎回見てもらう） | 中（記録が貯まると楽しい） | 高（出荷前に毎回通す） |
| **差別化** | AI×辛口という新しさ | ポートフォリオ的蓄積 | CI/CDのデザイン版 |
| **継続動機** | 上達の実感 | 記録の蓄積 | 合格の達成感 |
| **最大のリスク** | AIの指摘精度 | 記録の面倒さ | 基準の信頼性 |
| **必要なAI品質** | 高（具体的提案が命） | 中（差分検出が中心） | 中（チェック項目ベース） |
